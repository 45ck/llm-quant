## System Context

### The llm-quant System

llm-quant is a systematic macro trading system managing a $100,000 paper trading portfolio across 39 tradeable assets spanning six asset classes:

- **US Equity ETFs (17)**: SPY, QQQ, IWM, DIA, VTI, IVV, XLK, XLF, XLE, XLV, XLI, XLY, XLP, XLU, XLRE, XLB, XLC
- **International Equity (3)**: VGK, EEM, EFA
- **Fixed Income (6)**: TLT, IEF, SHY, LQD, HYG, TIP
- **Commodities (3)**: GLD, SLV, USO
- **Crypto (5)**: BTC-USD, ETH-USD, SOL-USD, XRP-USD, ADA-USD
- **Forex (5)**: EURUSD=X, GBPUSD=X, USDJPY=X, AUDUSD=X, USDCHF=X

The system runs Python 3.12 with Polars DataFrames for computation and DuckDB for persistent storage. Technical indicators are computed in `src/llm_quant/data/indicators.py` and currently include SMA(20/50/200), Bollinger Bands(20,2), RSI(14), MACD(12,26,9), and ATR(14) -- computed using Wilder's smoothing with alpha=1/14. Market data is stored in the `market_data_daily` table with schema `(symbol, date, open, high, low, close, volume, adj_close, sma_20, sma_50, rsi_14, macd, macd_signal, macd_hist, atr_14)`.

The decision pipeline flows: `scripts/build_context.py` (fetches data, computes indicators, assembles JSON context) -> Claude LLM (regime assessment, signal generation with `target_weight` per asset) -> `scripts/execute_decision.py` (risk checks, execution, persistence). The LLM outputs `TradeSignal(symbol, action, conviction, target_weight, stop_loss, reasoning)` where `target_weight` is the desired portfolio weight as a fraction of NAV.

Performance targets: Sharpe > 0.8, max drawdown < 15%, Sortino > 1.0, Calmar > 0.5.

### Current Sizing Approach

Position sizing in llm-quant is currently determined by two mechanisms:

1. **LLM-proposed target weight**: The Claude LLM decides a `target_weight` (e.g., 0.05 for 5% of NAV) for each `TradeSignal`. This is a judgment call, not a formula.

2. **Flat risk limits as caps**: The `RiskManager` enforces hard ceilings via `check_position_weight()`:
   - Equity ETFs: max 10% of NAV per position (`max_position_weight = 0.12` in config)
   - Crypto: max 5% of NAV per position (`crypto_max_position_weight = 0.05`)
   - Forex: max 8% of NAV per position (`forex_max_position_weight = 0.08`)

There is no volatility normalization. The LLM may choose 5% for BTC-USD and 10% for TLT, but these are treated as equivalent risk allocations by the risk system. They are not.

### The Critical Gap This Research Addresses

**There is no volatility-normalized sizing. Position weights are flat percentages of NAV with no adjustment for the dollar risk each position contributes.**

The consequence is dramatic. Consider the typical daily ATR as a percentage of price:

| Asset | Typical ATR(14) % of Price | $10,000 Position Daily $ Risk |
|-------|---------------------------|-------------------------------|
| SHY | 0.05% | $5 |
| TLT | 0.70% | $70 |
| SPY | 0.90% | $90 |
| GLD | 1.00% | $100 |
| QQQ | 1.20% | $120 |
| USO | 2.00% | $200 |
| BTC-USD | 3.50% | $350 |
| ETH-USD | 4.50% | $450 |
| SOL-USD | 6.00% | $600 |

BTC-USD at the 5% weight cap contributes **$175 of daily dollar risk** ($5,000 * 3.50%), while TLT at the 10% weight cap contributes only **$70 daily dollar risk** ($10,000 * 0.70%). BTC contributes 2.5x the risk at half the weight. SOL-USD at 5% contributes **$300 daily dollar risk** -- more than 4x what TLT at 10% contributes.

The portfolio's realized volatility is dominated by a small number of high-ATR positions, regardless of their weight. This means:
- Stop-loss distances are arbitrary (5% for equity, 15% for crypto) rather than calibrated to realized volatility
- The 15% max drawdown target is uncontrolled because the portfolio's risk profile depends on which assets happen to be held, not their weights
- The LLM has no quantitative framework for choosing `target_weight` -- it guesses

## Research Question

Does replacing flat weight-cap sizing with ATR-scaled equal-dollar-risk sizing reduce realized portfolio volatility by 30-40% while maintaining comparable Sharpe ratio?

Specifically:
1. What is the actual risk contribution of each asset class under the current flat-weight scheme?
2. Does equal-dollar-risk sizing (position_size = risk_budget / ATR_pct / price) produce materially lower portfolio volatility?
3. What is the optimal daily risk budget per position that maximizes Sharpe while respecting the 15% max drawdown constraint?
4. How should ATR-based sizing interact with the existing weight caps (replace them, supplement them, or operate as a parallel constraint)?

## Hypothesis

BTC-USD at 5% weight contributes approximately 10-15x the daily dollar risk of TLT at 10%. Equal-dollar-risk sizing, where each position is sized so that `position_notional * ATR_pct = constant_risk_budget`, will:

- **Reduce annualized portfolio volatility by 30-40%** relative to the current flat-weight scheme, because high-ATR assets (crypto, commodities) will receive significantly smaller notional allocations
- **Reduce absolute annualized returns by no more than 10-15%**, because the reduction comes primarily from downsizing positions in high-volatility assets that contribute noise, not from removing exposure to return-generating factors
- **Preserve or improve Sharpe ratio** (within +/- 10%), because the volatility reduction exceeds the return reduction
- **Reduce max drawdown by 25-35%**, because the largest drawdown contributors (high-ATR assets during stress) are structurally smaller
- **Produce a more stable risk profile** where the ex-ante daily portfolio dollar risk is predictable and controllable, rather than being an emergent property of which assets the LLM happens to select

The formula for equal-dollar-risk sizing:

```
daily_risk_budget_per_position = NAV * target_portfolio_risk / max_positions
target_weight = daily_risk_budget_per_position / (ATR_pct * price) * price / NAV
             = daily_risk_budget_per_position / (ATR_pct * NAV)
```

Where `target_portfolio_risk` is the total daily dollar risk budget as a fraction of NAV (e.g., 0.005 = 0.5% of NAV per day across all positions), and `max_positions` is the maximum number of concurrent positions (e.g., 15).

3. **Compute hypothetical equal-dollar-risk weights** for each date:
   ```
   equal_risk_weight[i] = (target_risk_per_position / atr_pct[i]) / NAV
   ```
   Where `target_risk_per_position` is calibrated so that the sum of all weights equals a reasonable gross exposure (e.g., 60-80% of NAV). Compare these weights against the flat weight caps.

### Phase 2: Sizing Function Design (Week 2-3)

5. **Design the `compute_atr_position_size()` function** with the following logic:

   ```python
   def compute_atr_position_size(
       symbol: str,
       price: float,
       atr_14: float,
       nav: float,
       risk_budget_per_position: float,  # e.g., 0.0033 = 0.33% NAV
       min_weight: float = 0.01,         # floor at 1% to avoid dust positions
       max_weight: float = 0.10,         # hard cap (may differ by asset class)
   ) -> float:
       """Return target weight as fraction of NAV."""
       if atr_14 <= 0 or price <= 0:
           return 0.0
       atr_pct = atr_14 / price
       raw_weight = risk_budget_per_position / atr_pct
       return max(min_weight, min(raw_weight, max_weight))
   ```

6. **Calibrate `risk_budget_per_position`**: Grid search over values from 0.001 (0.1% NAV per day per position) to 0.010 (1.0% NAV per day per position) in steps of 0.001. For each value, compute:
   - Resulting weights for each asset
   - Total gross exposure
   - Expected daily portfolio dollar risk (sum of individual position risks, ignoring diversification)
   - Expected daily portfolio dollar risk (accounting for correlation structure from Research 3.1 if available, otherwise ignore)

7. **Handle edge cases**:
   - **ATR = 0 or near-zero** (e.g., SHY during calm periods): Cap weight at `max_weight` to avoid infinite position size
   - **ATR spike** (e.g., VIX event causes ATR to triple overnight): Position size shrinks automatically, which is the desired behavior, but may trigger excessive rebalancing
   - **New positions with stale ATR**: If ATR was computed from stale data (e.g., weekend for crypto), use a lookback buffer
   - **Very high ATR assets** (SOL-USD with 6%+ daily ATR): Weight may be floored at `min_weight` most of the time, making the position effectively symbolic

11. **ATR-based stop-loss distances**: Replace the fixed percentage stop-losses (5% equity, 15% crypto, 3% forex) with ATR-multiple stops:
    ```
    stop_loss_price = entry_price - (N * ATR_14)
    ```

    Where N is the ATR multiplier. Test N = {1.5, 2.0, 2.5, 3.0} for each asset class.

13. **Interaction with position sizing**: When ATR increases (volatility spike), ATR-based sizing shrinks the position AND the ATR-based stop moves further from entry. Verify that the dollar risk per position (position_size * stop_distance) remains approximately constant under various ATR regimes.

### Phase 5: Integration Design (Week 5-6)

14. **Design the integration with the LLM decision pipeline**: The LLM currently proposes `target_weight` as a free parameter. Under ATR sizing, the LLM should instead propose a **conviction-scaled risk allocation** (e.g., "allocate 1x risk budget to SPY" for medium conviction, "allocate 1.5x risk budget to SPY" for high conviction), and the sizing function translates this into an actual weight using ATR.

15. **Design the interaction with existing risk checks**: ATR-sized positions must still pass all 8 existing checks. Verify that:
    - `check_position_weight()` is satisfied because ATR-sized weights are clamped to max_weight
    - `check_position_size()` is satisfied because incremental trade notional respects max_trade_size
    - `check_gross_exposure()` is satisfied because total ATR-sized weights sum to a controlled level
    - The sizing function can be used to set `target_weight` on the `TradeSignal` before it enters the risk gate

16. **Rebalancing triggers**: ATR changes daily. If positions are sized to ATR at entry but ATR drifts significantly, the risk budget per position becomes misaligned. Define rebalancing rules:
    - Rebalance when `current_atr_pct / entry_atr_pct > 1.5` (ATR increased 50%) or `< 0.67` (ATR decreased 33%)
    - Cap rebalancing frequency at once per week per position to avoid excessive turnover

## Literature Review

### Core Papers

1. **Carver, R. (2015).** *Systematic Trading: A Unique New Method for Designing Trading and Investing Systems.* Harriman House, Chapters 9 & 11.
   - **Extract**: Chapter 9 ("Position Sizing") introduces the concept of volatility-standardized position sizing where each instrument is sized to contribute an equal amount of risk. Carver uses a "risk target" (annualized standard deviation target for the portfolio) and scales each position by its forecast volatility. Chapter 11 ("Portfolio Construction") discusses how to combine positions using correlation-aware risk budgets.
   - **Application**: Carver's framework directly applies to our problem. His formula `position_size = capital * risk_target / (instrument_risk * price)` is the theoretical foundation for `compute_atr_position_size()`. Use ATR as a proxy for `instrument_risk` (Carver uses annualized standard deviation, but daily ATR * sqrt(252) is a reasonable approximation).

2. **Baltas, N. & Kosowski, R. (2013).** "Momentum Strategies in Futures Markets and Trend-Following Funds." *SSRN Working Paper*.
   - **Extract**: Volatility-scaled momentum strategies (where position size is inversely proportional to recent volatility) produce higher Sharpe ratios than unscaled strategies. The improvement comes from two sources: (1) reducing outsized losses from high-volatility instruments, and (2) increasing exposure to low-volatility instruments that have a more favorable return-to-risk ratio.
   - **Application**: Their empirical evidence supports our hypothesis that ATR-scaled sizing improves Sharpe. Extract their reported Sharpe improvement magnitudes (typically 20-40% higher) as a calibration benchmark.

3. **Harvey, C.R., Rattray, S., & Van Hemert, O. (2021).** "Strategic Risk Management: Designing Portfolios and Managing Risk." *Journal of Portfolio Management*, 47(5).
   - **Extract**: Risk budgeting frameworks where portfolio risk is decomposed into additive contributions from each position. The risk contribution of position i equals `w_i * (Sigma * w)_i` where Sigma is the covariance matrix and w is the weight vector. Equal risk contribution (ERC) portfolios equalize these marginal contributions.
   - **Application**: ATR equal-dollar-risk is a simplified version of ERC that ignores correlations. Benchmark our approach against full ERC to determine whether the simpler ATR-only method captures most of the benefit without requiring a covariance matrix.

4. **Moskowitz, T.J., Ooi, Y.H., & Pedersen, L.H. (2012).** "Time Series Momentum." *Journal of Financial Economics*, 104(2), 228-250. Section 4 on volatility scaling.
   - **Extract**: Section 4 demonstrates that scaling positions by the inverse of trailing volatility (ex-ante volatility targeting) significantly improves risk-adjusted returns for time-series momentum strategies across 58 instruments. The benefit is robust across asset classes and time periods.
   - **Application**: Their volatility-scaling methodology is directly applicable. They use a 60-day exponentially weighted standard deviation; we use ATR(14). Extract their reported improvement in Sharpe ratio from volatility scaling to benchmark our expected improvements.

### Supplementary Papers

5. **Maillard, S., Roncalli, T., & Teiltche, J. (2010).** "The Properties of Equally Weighted Risk Contribution Portfolios." *Journal of Portfolio Management*, 36(4), 60-70.
   - **Extract**: Formal properties of ERC portfolios. ERC lies between equal-weight and minimum-variance on the efficient frontier. ERC is more robust to estimation error than minimum-variance because it does not require expected return estimates.
   - **Application**: Provides the theoretical foundation for why equal-risk-contribution is a good default sizing scheme when expected returns are uncertain (as they are when the LLM is making judgment calls).

6. **Bouchaud, J.P. & Potters, M. (2003).** *Theory of Financial Risk and Derivative Pricing.* Cambridge University Press, Chapter 10.
   - **Extract**: Analysis of how volatility clustering (GARCH effects) impacts position sizing. When volatility is mean-reverting, sizing to current volatility is suboptimal because current high volatility predicts lower future volatility. A blend of current and long-term average volatility may be more appropriate.
   - **Application**: Consider using a blended ATR estimate (e.g., 70% ATR_14 + 30% ATR_63) to avoid excessive position size changes during volatility spikes. Test whether this blend reduces turnover without sacrificing risk control.

## Evaluation Criteria

### Success Thresholds

| Criterion | Threshold | Rationale |
|-----------|-----------|-----------|
| Portfolio volatility reduction | >= 25% | Primary objective. Must produce a meaningful reduction in realized vol. |
| Sharpe ratio preservation | >= -10% relative to baseline | ATR sizing should not significantly impair risk-adjusted returns. Ideally improves Sharpe. |
| Max drawdown reduction | >= 15% | Smaller high-ATR positions should reduce tail losses. |
| Risk concentration improvement | Top 3 positions <= 50% of total dollar risk | Currently, top 3 high-ATR positions may contribute 70-80% of risk. Target more balanced risk. |
| Turnover increase | <= 50% above baseline | ATR-driven rebalancing must not create excessive turnover. |
| Daily dollar risk predictability | Coefficient of variation <= 0.30 | Daily portfolio risk should be stable, not wildly variable based on which assets are held. |

### Failure Conditions

- If Sharpe ratio decreases by more than 15% under any reasonable `risk_budget_per_position`, the volatility reduction is coming at too high a return cost.
- If ATR-driven rebalancing produces more than 2x baseline turnover, the implementation needs dampening (wider rebalance thresholds or slower ATR averaging).
- If high-ATR assets (crypto, USO) are consistently floored at `min_weight` (1%), the sizing function effectively excludes them from the portfolio, which may not be the intent. If this occurs, consider whether the correct response is to accept it (those assets are too volatile for the strategy's risk budget) or to adjust parameters.

## Anti-Overfitting Warnings

### Specific Pitfalls for This Research

1. **Volatility paradox -- sizing to historical ATR assumes ATR is predictive**: ATR(14) reflects the last 14 days. If volatility regime-shifts (e.g., calm to crisis in 1 day), the ATR-based size is wrong for the new regime. **Mitigation**: Test sensitivity to ATR window (7, 14, 21, 42 days). Consider using the maximum of short-term and long-term ATR for conservative sizing: `effective_atr = max(atr_14, atr_63)`.

2. **Optimizing `risk_budget_per_position` on historical data is curve-fitting**: A budget that maximizes Sharpe over 2020-2024 may not be optimal for 2025-2026. **Mitigation**: Use walk-forward optimization. Calibrate on years 1-3, validate on year 4, test on year 5. Report out-of-sample performance. Also test sensitivity: if Sharpe is flat across a range of budgets (e.g., 0.002-0.005), that is more robust than if it peaks sharply at one value.

3. **ATR-based sizing penalizes high-volatility assets that may have the best risk-adjusted returns**: Crypto and commodities have high ATR but may also have high expected returns (momentum, carry). Sizing them smaller reduces portfolio exposure to these return drivers. **Mitigation**: Compare Sharpe ratios per asset class under flat vs. ATR sizing. If high-ATR assets have higher *risk-adjusted* returns (Sharpe), the ATR sizing is overly conservative for those assets. Consider conviction-scaled risk budgets (high conviction = 1.5x budget) to allow the LLM to override the default sizing for high-Sharpe opportunities.

4. **Turnover from ATR-driven rebalancing can eat returns via transaction costs**: Even though this is paper trading with no explicit transaction costs, excessive turnover is a red flag for real-world viability. **Mitigation**: Track hypothetical round-trip transaction costs (0.05% per trade for equity ETFs, 0.10% for crypto, 0.02% for forex) and deduct from returns. If turnover drag exceeds 1% annualized, tighten rebalance thresholds.

5. **Interaction with LLM discretion**: The LLM currently has discretion to set `target_weight`. ATR sizing removes this discretion. **Mitigation**: Measure the correlation between LLM-proposed weights and ATR-implied weights to determine whether the LLM is already doing implicit vol-adjustment.

6. **Equal-dollar-risk is not equal-risk-contribution**: True ERC accounts for correlations; two positions with identical ATR but correlation = 0.95 contribute nearly double the risk of two uncorrelated positions. **Mitigation**: Acknowledge this limitation. If Research 3.1 provides a correlation surface, extend to full marginal-risk-contribution sizing.

7. **Multi-dimensional optimization risk**: Jointly optimizing risk_budget + stop_multiplier + ATR_window creates a 3D search with high overfitting risk. **Mitigation**: Fix one dimension at a time. First optimize risk_budget with fixed stop_multiplier=2.0 and ATR_window=14. Then optimize the others sequentially.
